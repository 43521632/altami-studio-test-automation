# Как создать UI-кейс для любой системы (пошаговый разбор на примере Altami Studio)

Практическое руководство: как я находил координаты, чем кликал, какие клавиши
использовал, как проверял надписи и окна, и как запустить все тесты. По этой
инструкции вы соберёте кейс для **любой** гостевой ОС (Astra, Windows, macOS),
а не только повторите готовый.

Дополняет [writing-tests.md](writing-tests.md) (базовые понятия, фикстуры,
логин, пороги SSIM). Здесь — именно **процесс разработки кейса руками**.

---

## 0. Модель, которую надо держать в голове

* **Чёрный ящик через QMP.** Внутри гостя ничего не установлено. Мы двигаем
  мышь, жмём клавиши и снимаем экран снаружи, через второй QMP-сокет. Гость об
  автоматизации не знает.
* **Координаты — в пикселях гостя**, отсчёт от левого верхнего угла, и они
  привязаны к **видеорежиму**. Координаты для 1920×1200 не подойдут для
  1280×800. У Astra экран входа (fly-dm) 1280×800, а рабочий стол 1920×1200 —
  это разные наборы координат.
* **Одна ВМ — один процесс pytest, тесты идут последовательно и делят экран.**
  Состояние после одного теста достаётся следующему. Либо возвращайте UI в
  исходное состояние, либо стройте кейс так, чтобы порядок не мешал.
* **Проверяем не «весь экран», а статичную область** (region SSIM). Часы, даты,
  курсор ломают полноэкранное сравнение.

## 1. Инструмент разведки: `scripts/ui_probe.py`

Это ручной драйвер гостевого UI. Им вы вслепую не работаете: сняли скриншот →
посмотрели глазами → нашли координату → проверили клик. Какую ВМ вести, задаёт
`VM_ID` (как в тестах).

```bash
# всё запускаем из корня репозитория, интерпретатором из venv
export VM_ID=astra          # или windows / macos

./venv/bin/python scripts/ui_probe.py res                  # разрешение гостя
./venv/bin/python scripts/ui_probe.py shot desktop         # скриншот -> путь к PNG
./venv/bin/python scripts/ui_probe.py move 960 600         # телепорт указателя
./venv/bin/python scripts/ui_probe.py glide 22 1174        # ПЛАВНОЕ перемещение
./venv/bin/python scripts/ui_probe.py click 22 1174        # клик
./venv/bin/python scripts/ui_probe.py glideclick 22 1174   # плавно навести и кликнуть
./venv/bin/python scripts/ui_probe.py dclick 100 200       # двойной клик
./venv/bin/python scripts/ui_probe.py key ret              # клавиша
./venv/bin/python scripts/ui_probe.py key ctrl alt t       # комбинация
./venv/bin/python scripts/ui_probe.py type "echo hi"       # набор ASCII
./venv/bin/python scripts/ui_probe.py boot                 # старт ВМ + логин
```

Последняя позиция указателя запоминается между вызовами (файл в `TMPDIR`),
поэтому `glide` ведёт курсор от предыдущей точки, как настоящая мышь.

> Требования к окружению (второй QMP-сокет, USB-планшет, права на каталог
> скриншотов, правило AppArmor) — в README. Без них клики уходят мимо, а
> скриншоты не создаются.

## 2. Запустить ВМ и войти в систему

Отдельно поднимать ВМ для написания кейса не обязательно, но при разведке
удобно сделать это один раз руками:

```bash
VM_ID=astra ./venv/bin/python scripts/ui_probe.py boot
```

Команда `boot` стартует домен, ждёт стабилизации экрана (прокси «ОС
загрузилась») и выполняет автологин ровно так же, как фикстуры в тестах. Вход
описан в `config/vms_config.yaml → vms.<vm>.login` (последовательность кликов и
набора), пароль — в `.env` (`<VM_ID>_LOGIN_PASSWORD`). Кириллицу в пароле QMP
не наберёт — только латиница.

После `boot` на экране готовый рабочий стол — с него и начинается кейс.

## 3. Снять скриншот и определить координаты

```bash
VM_ID=astra ./venv/bin/python scripts/ui_probe.py shot desktop
# SHOT=/var/lib/libvirt/screenshots/astra/probe_desktop.png
```

Скриншоты лежат **вне репозитория**, в `/var/lib/libvirt/screenshots/<vm>/` —
так надо, потому что PNG пишет процесс QEMU (пользователь `libvirt-qemu`), а не
мы.

**Как получить координату элемента:** откройте PNG в любом просмотрщике или
редакторе, где под курсором видны координаты пикселя (GIMP, Krita, даже
предпросмотр в IDE), и наведитесь на центр нужной кнопки. Это и есть `(x, y)` в
пикселях гостя.

Практика:
* Целитесь в **центр** элемента, а не в край — попадание надёжнее.
* Кнопки на панели задач и в трее стабильны; элементы внутри окон — по факту.
* Если элемент появляется не сразу (окно запускается) — снимите несколько
  кадров подряд и выберите тот, где он уже отрисован.

## 4. Как кликать и наводить: `move`/`click` против `glide` — ГЛАВНЫЙ УРОК

QMP умеет два вида перемещения:

| Способ | Что делает | Когда работает |
|---|---|---|
| `move` / `click` | **телепорт**: одно событие абсолютной позиции | панель задач, простые кнопки диалогов |
| `glide` / `glideclick` | **плавное движение** мелкими шагами | меню, подменю, наведение-раскрытие |

**Почему это критично.** Меню fly (кнопка «Пуск» → категории → ярлыки)
раскрываются и подсвечиваются по событию **движения** мыши. Одиночный телепорт
Qt как движение не распознаёт: подменю не выезжает, пункт не подсвечивается,
клик по нему не срабатывает — и весь сценарий молча идёт по пустому рабочему
столу. На этом тест Altami падал трижды, пока перемещение по меню не заменили на
`glide`.

Правило:
* по **пунктам меню и наведению** — только `glide` / `glideclick`;
* по **обычным кнопкам** (в диалогах, на панели задач) — достаточно `click`, но
  `glideclick` надёжнее и не мешает.

Проверяйте руками: навели `glide` на пункт → `shot` → на скриншоте пункт должен
быть **подсвечен**. Не подсвечен — клик не пройдёт.

## 5. Клавиатура (qcode)

Имена клавиш — это `QKeyCode` из `qapi/ui.json`, а не привычные подписи:

| Клавиша | qcode | Клавиша | qcode |
|---|---|---|---|
| Enter | `ret` | Пробел | `spc` |
| Esc | `esc` | Tab | `tab` |
| Backspace | `backspace` | Delete | `delete` |
| Стрелки | `up` `down` `left` `right` | Win/Meta | `meta_l` |
| Ctrl / Alt / Shift | `ctrl` `alt` `shift` | F1–F12 | `f1`…`f12` |

```bash
./venv/bin/python scripts/ui_probe.py key esc
./venv/bin/python scripts/ui_probe.py key ctrl alt t
./venv/bin/python scripts/ui_probe.py type "hello world"   # ASCII, авто-Shift для A-Z и !@#$
```

> **Кириллица не набирается.** Раскладка qcode американская, `type` умеет только
> ASCII. Русский текст вводите через буфер обмена гостя или не вводите вовсе.

## 6. Проверки: сравнение по области (region SSIM)

Полноэкранное сравнение (`assert_screen`) ломается, как только в кадре есть
меняющиеся зоны: часы в трее, список сессий с датами, курсор, анимации. Поэтому
надписи и окна проверяем **по вырезанной статичной области** — это добавлено в
`tests/base_tests.py`:

```python
await self.assert_region("altami_demo_banner", (668, 712, 1256, 878))
# box = (left, top, right, bottom) в пикселях гостя
```

Три метода на выбор:
* `assert_region(name, box)` — вырезать область, сверить с эталоном, упасть при
  расхождении (SSIM ≤ порога);
* `compare_region(name, box)` — то же, но без падения (вернёт результат);
  удобно проверять, что окно **исчезло** — тогда область НЕ должна совпасть:
  ```python
  gone = await self.compare_region("altami_restore_title", RESTORE_TITLE_BOX)
  assert not gone.passed, "окно не закрылось"
  ```
* `capture_region(name, box)` — просто вырезать (для отладки).

**Как выбрать область:**
1. Возьмите статичный участок: заголовок окна, ряд кнопок, полосу с надписью,
   тулбар. Избегайте часов, дат, курсора, анимаций.
2. Определите `box` по скриншоту тем же способом, что и координаты (углы
   прямоугольника).
3. Проверьте вырезку глазами — скриптом на Pillow:
   ```python
   from PIL import Image
   Image.open("/var/lib/libvirt/screenshots/astra/probe_x.png") \
        .crop((668, 712, 1256, 878)).save("/tmp/check.png")
   ```
   Откройте `/tmp/check.png`: в области должно быть ровно то, что проверяете, без
   лишнего.

**Эталон.** При первом прогоне отсутствующий эталон
`baseline/<vm>/<name>.png` создаётся из текущей вырезки, и тест проходит —
то есть первый прогон ничего не проверяет. Поэтому эталон осмысленно
**засеивают** заранее из заведомо правильного кадра:

```python
from PIL import Image
from pathlib import Path
Image.open("/var/lib/libvirt/screenshots/astra/probe_banner.png") \
     .crop((668, 712, 1256, 878)).convert("RGB") \
     .save(Path("baseline/astra/altami_demo_banner.png"))
```

`box` при засеве и в тесте должен быть **один и тот же** — иначе размеры не
сойдутся. Эталоны в git не хранятся (они локальные), это норма для проекта.

Порог SSIM задаётся в `config/vms_config.yaml → test_settings.screenshot_comparison.threshold`
(по умолчанию `0.99`).

## 7. Надёжность: «проверил — повторил» и парковка курсора

QMP-ввод изредка теряет первое событие, а меню бывают капризны. Поэтому каждый
критичный шаг **подтверждают проверкой и повторяют при неудаче**, а не жмут
вслепую. В тесте Altami так сделаны три шага:

```python
# открыть меню и убедиться, что колонка категорий появилась; иначе кликнуть ещё
for attempt in range(1, 4):
    await self.glide_click(*START_STAR)
    await asyncio.sleep(1.5)
    await self._park_mouse()
    if (await self.compare_region("altami_menu_open", MENU_OPEN_BOX)).passed:
        return
# ... аналогично: подменю раскрылось? приложение запустилось (меню закрылось)?
```

**Парковка курсора.** Перед снимком области уводите указатель в нейтральную
точку (`_park_mouse` → угол вне проверяемых областей), иначе курсор попадёт в
кадр и собьёт SSIM.

## 8. Рецепт: как именно собирался кейс Altami Studio

Пошагово, чтобы повторить логику для своей системы:

1. **Разведка.** `ui_probe boot` → `shot desktop`. Нашёл звезду «Пуск» в углу
   панели задач: `(22, 1174)`.
2. **Меню.** `glideclick 22 1174` → `shot` → меню открылось. Снял координату
   «Научные»: `(250, 873)`.
3. **Подменю.** `glide 250 873` → `shot` → выехало подменю с «Altami Studio».
   Здесь и вскрылось, что **телепорт не раскрывает подменю, а `glide`
   раскрывает**. Координата ярлыка: `(560, 715)`.
4. **Запуск.** `glide 560 715` (пункт подсветился) → `click 560 715` → пошёл
   splash-баннер «ALTAMI STUDIO / Демо-версия / v.4.1».
5. **Области для проверок** — вырезал из хороших кадров и засеял эталоны:
   * `altami_demo_banner (668,712,1256,878)` — надпись «Демо-версия» (главная
     проверка кейса);
   * `altami_restore_title (744,386,1162,416)` — заголовок окна «Восстановить
     состояние» (окно есть);
   * `altami_app_toolbar (0,54,1132,96)` — тулбар приложения (UI не поломался);
   * `altami_menu_open`, `altami_nauchnye_submenu` — для проверок «меню
     открылось / подменю раскрылось».
6. **Кнопки диалогов.** «Закрыть» `(1147, 563)`, «Пропустить» `(1101, 634)` —
   через `glideclick`.
7. **Тайминги** взял из сценария: ~8 с на splash, ~4 с на окно восстановления —
   фиксированными паузами `asyncio.sleep`, потому что баннер держится до
   «Закрыть», а не мелькает.

Результат — `tests/Astra/test_astra_altami_studio.py`, один тест на весь поток с
проверками по областям и повторами на каждом капризном шаге.

## 9. Структура файла теста

```python
import asyncio
import pytest
from tests.base_tests import BaseVMTest

# координаты и области — константами вверху, с пометкой откуда сняты
START_STAR = (22, 1174)          # проверено на живой ВМ, режим 1920x1200

@pytest.mark.astra      # ОС: astra | windows | macos
@pytest.mark.ui         # тип: ui | system | smoke  (маркеры строгие!)
class TestAstraAltamiStudio(BaseVMTest):
    vm_id = "astra"     # тест сам пропустится, если прогон идёт на другой ВМ

    async def test_...(self):        # только async, await на каждом действии
        await self.glide_click(*START_STAR)
        await asyncio.sleep(1.5)     # пауза между действием и проверкой
        await self.assert_region("name", (l, t, r, b))
```

Чек-пункты:
* файл `tests/<Os>/test_*.py`, класс наследует `BaseVMTest`, поле `vm_id`;
* маркеры ОС + типа обязательны и должны быть зарегистрированы в `pytest.ini`
  (незнакомый маркер уронит прогон);
* методы — `async def`, `await` на каждом шаге;
* между действием и проверкой — пауза (`asyncio.sleep`) или
  `await self.session.wait_until_screen_stable(...)`;
* верните UI в исходное состояние в конце **или** осознанно оставьте как есть,
  если следующий кейс продолжает с этой точки (как у Altami — приложение
  остаётся открытым намеренно).

Доступные примитивы `self` (см. `tests/base_tests.py`): `click`, `double_click`,
`glide`, `glide_click`, `type_text`, `press`, `capture`, `assert_screen`,
`capture_region`, `compare_region`, `assert_region`; всё низкоуровневое — через
`self.qmp`.

## 10. Как запустить тесты

Всё из корня репозитория, интерпретатором из venv. Пароли логина берутся из
`.env`, поэтому его надо подгрузить в окружение.

```bash
set -a; . ./.env; set +a          # подставить <VM>_LOGIN_PASSWORD и т.п.

# ВСЕ тесты Astra
VM_ID=astra ./venv/bin/pytest tests/Astra/ -v

# один файл
VM_ID=astra ./venv/bin/pytest tests/Astra/test_astra_altami_studio.py -v

# один тест по имени
VM_ID=astra ./venv/bin/pytest tests/Astra/ -k altami -v

# с живым логом в консоль (удобно при отладке)
VM_ID=astra ./venv/bin/pytest tests/Astra/ -v --log-cli-level=INFO
```

Windows — так же, с `VM_ID=windows` и `tests/windows/`. Интерактивное меню
запуска/остановки ВМ: `./venv/bin/python run_tests.py --menu`.

> **Тёплая ВМ загружается «медленно».** Если ВМ уже залогинена, фикстура входа
> ждёт весь таймаут гритера, прежде чем понять, что вход уже выполнен (~2 мин).
> На холодном старте (ВМ выключена) экран входа опознаётся сразу и прогон
> быстрее.

**Где смотреть результат:**
* лог — `logs/test_run.log`, `logs/pytest.log`;
* скриншоты (в т.ч. `FAILED_*` при падении) —
  `/var/lib/libvirt/screenshots/<vm>/`;
* дифф-картинки `[эталон | текущий | различия]` —
  `screenshots/diff/<vm>_<name>_<время>_diff.png`;
* отчёты — `reports/<vm>/`.

При падении сравнения по области смотрите дифф: средняя панель — что реально
было на экране в момент проверки. Именно так выяснилось, что приложение не
запускалось (в области был рабочий стол, а не баннер).

## 11. Чек-лист кейса для новой системы

1. `VM_ID=<vm> ui_probe boot` — поднять ВМ и войти.
2. `shot` за `shot`: пройти сценарий руками, на каждом шаге снимать экран и
   снимать координаты элементов.
3. По меню — `glide`/`glideclick`, проверяя подсветку; по кнопкам — `click`
   хватает.
4. Выбрать статичные области для проверок, вырезать и **засеять эталоны**.
5. Написать тест: координаты и области — константами, каждый капризный шаг — с
   проверкой и повтором, курсор парковать перед снимком.
6. Прогнать, разобрать дифф при падении, поправить координаты/паузы.
7. Решить судьбу финального состояния: вернуть UI назад или оставить для
   следующего кейса.
